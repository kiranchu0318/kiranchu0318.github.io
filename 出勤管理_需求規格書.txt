出勤管理（出勤管理.html）需求規格書
================================================================================
文件版本：v1.0
文件日期：2026-06-22
對應頁面：出勤管理.html（HR 員工出勤管理系統）
撰寫範圍：本頁前端（畫面、互動、資料呈現）與後端（API、資料表、業務邏輯）規格

--------------------------------------------------------------------------------
0. 文件目的與範圍
--------------------------------------------------------------------------------
0.1 目的
    定義「出勤管理」頁面之功能需求，作為前端、後端開發與測試之共同依據。
    目前頁面為靜態原型（HTML + Tailwind CDN + 前端 mock 資料），本規格書同時
    描述「現況（原型）」與「正式系統需具備之後端規格」，供後續串接實作。

0.2 角色定位
    本頁為「人資（HR）／管理者視角」之出勤資料檢視與審核管理頁，
    以查詢、檢視、匯出為主；員工自助申請於其他頁面（如假期申請、補卡申請）完成。

0.3 名詞定義
    - 出勤紀錄：員工每日上/下班打卡資料。
    - 班別（shift）：同一日可能有多組上/下班時段（例：早班、晚班）。
    - 出勤狀況：當日相關的補卡／請假／銷假單據彙整（含審核狀態）。
    - 補卡：員工漏打卡後提出之補登申請。
    - 外出：上班時段中外出公務並於返回後登記之紀錄。
    - 地區 / 事業部 / 項目 / 部門：員工所屬之組織歸屬維度。

第一部分　前端規格（FRONTEND SPEC）

--------------------------------------------------------------------------------
1. 頁面整體
--------------------------------------------------------------------------------
1.1 技術現況
    - HTML5 + Tailwind CSS（CDN）+ Font Awesome 6.4.0 + 自訂 shared-ui.css。
    - 純前端互動，資料為頁內 JavaScript 陣列（attendanceDb / makeupDb / outingDb
      及請假資料於內嵌頁面）。正式版改由後端 API 提供。
    - 語系：繁體中文（zh-TW）。

1.2 版面
    - 頁首標題：「出勤管理」（ui-page-title）。
    - 主容器全寬自適應（w-full，無固定寬度上限），大螢幕撐滿、小螢幕內容不換行
      並由表格容器水平捲動。
    - 主色 primary = #4A90E2。

1.3 分頁（Tabs）— 共四個，預設顯示「出勤紀錄」
    順序：出勤紀錄 → 請假紀錄 → 補卡紀錄 → 外出紀錄
    - 切換僅顯示/隱藏對應面板（switchTab），不重新載入頁面。
    - 「請假紀錄」分頁以 iframe 內嵌「假期管理.html」整頁內容。
    - 各分頁各自擁有獨立的篩選列與列表。

--------------------------------------------------------------------------------
2. 共用 UI 元件
--------------------------------------------------------------------------------
2.1 篩選列（每個列表分頁皆有）
    - 採 grid 排版（lg 一行四欄）。
    - 「員工搜尋（姓名/工號）」文字框（即時 oninput 觸發篩選）。
    - 地區 / 事業部 / 項目 / 部門：可複選下拉（multi-select），未選=全部。
    - 各分頁專屬下拉與日期區間（見各分頁）。
    - 右下角操作鈕：
        「查詢」（ui-btn-primary，放大鏡圖示）→ 觸發該分頁篩選函式。
        「重置」（ui-btn-secondary，旋轉箭頭）→ 還原篩選並顯示 Toast。

2.2 列表卡片
    - 標題 + 筆數徽章（如「12 筆資料」）。
    - 右側色點圖例（各分頁不同，見下）。
    - 「匯出報表」按鈕（目前為 mock，僅顯示 Toast）。
    - 表格置於水平捲動容器；欄位不換行（whitespace-nowrap）。
    - 查無資料時顯示空狀態（圖示＋文字＋「清除篩選條件」）。
    - 底部分頁列（頁碼/上一頁/下一頁＋「顯示第 X 到 Y 筆，共 N 筆」文字）。
      ※ 現況分頁為靜態樣式，未實作分頁切換邏輯。

2.3 詳情彈窗（Modal）
    - 點列「操作 → 詳情」開啟，唯讀檢視，底部僅「關閉」鈕。
    - 進出場具淡入/縮放動畫；點遮罩或關閉鈕關閉。

2.4 Toast 提示
    - showToast(type, text)：success / info 等樣式，數秒後自動消失。

2.5 色點圖例對照
    - 出勤紀錄：綠色＝已審核、紅色＝已提交（對應「出勤狀況」單據顏色）。
    - 補卡紀錄：藍＝已提交、橘＝審批中、綠＝完成、紅＝駁回。
    - 外出紀錄：橘＝外出中、綠＝已返回。

--------------------------------------------------------------------------------
3. 分頁一：出勤紀錄
--------------------------------------------------------------------------------
3.1 篩選欄位
    - 員工搜尋（姓名/工號，模糊比對，不分大小寫）
    - 地區（複選）、事業部（複選）、項目（複選）、部門（複選）
    - 出勤狀態：全部 / 正常 / 異常
    - 出勤日期區間（起、迄）
    - 篩選為「條件 AND 組合」；日期以該筆出勤日期（date）落於區間判斷。

3.2 列表欄位（由左至右）
    (1) 工號
    (2) 姓名
    (3) 地區（徽章）
    (4) 事業部（徽章）
    (5) 項目（徽章）
    (6) 部門（徽章）
    (7) 日期
    (8) 上下班時間：支援同日多班別；每班「上」行＝上班、「下」行＝下班；
        補卡時間後標註「（補卡）」；未打卡顯示「--:--」（未打卡狀態紅字）。
    (9) 外出時間：彙整該員工當日所有外出，格式「外出-返回」，多筆逐列；
        尚未返回顯示「進行中」。
    (10) 打卡地點：對應各班別上/下班地點與打卡方式；過長以 … 省略，
         滑鼠移入（title）顯示完整地點。
    (11) 出勤狀況：彙整當日相關單據之單號（補卡、請假、銷假）。
         - 顏色規則：綠字＝已審核（單據狀態為「完成」）；紅字＝未審核（其餘狀態）。
         - 同日多張單據逐列堆疊；無相關單據顯示「-」。
         - 滑鼠移入顯示單據類型與審核狀態。
    (12) 狀態：由各班別節點狀態推導，可同時並存多個標籤：
         - 遲到（黃，ciState=遲到）
         - 早退（黃，coState=早退）
         - 缺卡（紅，ciState 或 coState=未打卡）
         - 若無任何異常則顯示「正常」（綠）。
    (13) 操作：「詳情」按鈕。

3.3 出勤詳情彈窗（檢視）
    - 員工資訊：工號、姓名、部門、職位。
    - 出勤資訊：出勤日期、打卡地點（多班別顯示「等 N 個班別」）、上班時間、
      下班時間、狀態徽章（同列表狀態邏輯）。
    - 打卡明細表：逐班別列出「上班打卡／下班打卡（第 N 班）」、打卡時間、
      地點、打卡方式、節點狀態徽章。
    - 備註：顯示 HR 備註，無則顯示「無備註」。

4. 分頁二：請假紀錄

4.1 現況：以 iframe 內嵌「假期管理.html」整頁（含其篩選、列表、詳情彈窗）。

5.3 補卡詳情彈窗：員工資訊；補卡資訊（申請日期/補卡日期/時段/補卡時間/補卡原因）；
    種類說明；審批備註（無則顯示「尚無審批備註」）。

7.（前端）互動與非功能需求
--------------------------------------------------------------------------------

8. API 規格（建議 RESTful，回傳 JSON，UTF-8）
--------------------------------------------------------------------------------
通則：
    - 所有清單 API 支援分頁（page、pageSize）、排序、與下列篩選參數。
    - 多選參數以陣列傳遞（region[]、bu[]、item[]、dept[]）。
    - 日期格式 YYYY-MM-DD；時間 HH:mm；時間戳 YYYY-MM-DD HH:mm:ss。
    - 權限：需登入；依角色（HR/主管/管理者）限制可見資料範圍（見 10）。

8.1 出勤紀錄清單
    GET /api/attendance/records
    Query：keyword（姓名/工號）、region[]、bu[]、item[]、dept[]、
           status（all|正常|異常）、dateStart、dateEnd、page、pageSize、sort
    Response（每筆）：
      id, name, dept, jobTitle, date,
      shifts: [ { ci, co, loc, method, ciState, coState, ciMakeup, coMakeup } ],
      status（正常|異常）, region, bu, item, note,
      attendanceFlags: { late:bool, earlyLeave:bool, missingPunch:bool }（後端可預算），
      relatedDocs: [ { type（補卡|請假|銷假）, docNo, status（完成|審批中|…）, approved:bool } ]
      （供前端「出勤狀況」欄使用）
    回傳同時含 total（總筆數）供分頁。

8.2 出勤紀錄詳情
    GET /api/attendance/records/{id}?date=YYYY-MM-DD
    Response：員工資訊、各班別打卡明細（每筆上/下班含時間/地點/方式/節點狀態）、備註。

8.3 補卡紀錄清單 / 詳情
    GET /api/attendance/makeups
    Query：keyword、region[]、bu[]、item[]、dept[]、reason、status、
           dateStart、dateEnd（依 makeupDate）、page、pageSize
    每筆：id, name, dept, appDate, makeupDate, period（上班/下班/上下班打卡）,
          time, reason, desc, status（已提交|審批中|完成|駁回）, note, region, bu, item
    GET /api/attendance/makeups/{makeupId} → 詳情。

8.4 外出紀錄清單 / 詳情
    GET /api/attendance/outings
    Query：keyword、region[]、bu[]、item[]、dept[]、status（外出中|已返回）、
           dateStart、dateEnd、page、pageSize
    每筆：id, name, dept, jobTitle, date, outTime, backTime, reason,
          status, region, bu, item
    GET /api/attendance/outings/{outingId} → 詳情。

8.5 請假紀錄清單 / 詳情（供「請假紀錄」分頁）
    GET /api/leaves
    Query：keyword、region[]、bu[]、item[]、dept[]、type、status、
           dateStart、dateEnd（依開始日/結束日重疊）、page、pageSize
    每筆：id, name, dept, jobTitle, startDate, endDate, type（特休/病假/事假/…）,
          days, reason, status（待審核|已核准|已駁回）, appDate, comment
    GET /api/leaves/{leaveId} → 詳情（含審批紀錄）。

8.6 匯出報表
    POST /api/attendance/export
    Body：reportType（出勤|補卡|外出）、上述對應篩選條件、format（xlsx|csv）
    Response：檔案下載連結或串流；大量資料採非同步產製＋下載通知。

8.7 篩選選項主檔
    GET /api/org/options → region、bu、item、dept 之可選清單（供多選下拉）。

9. 資料模型（建議資料表）
--------------------------------------------------------------------------------
9.1 employee（員工）
    emp_id(PK), name, dept_code, job_title, region_code, bu_code, item_code, status
9.2 attendance_record（出勤，逐班別）
    record_id(PK), emp_id(FK), work_date, shift_seq,
    check_in_time, check_out_time, check_in_loc, check_out_loc, punch_method,
    check_in_state（正常|遲到|未打卡|進行中…）,
    check_out_state（正常|早退|未打卡|進行中…）,
    check_in_makeup(bool), check_out_makeup(bool), note
    ※「整體狀態（正常/異常）」與「遲到/早退/缺卡」可由節點狀態即時推導或落地存放。
9.3 makeup_request（補卡申請）
    makeup_id(PK), emp_id(FK), apply_date, makeup_date, period, makeup_time,
    reason, description, status, approver_note, related_record_id(FK 可空)
9.4 outing_record（外出）
    outing_id(PK), emp_id(FK), out_date, out_time, back_time, reason, status
9.5 leave_request（請假/銷假）
    leave_id(PK), doc_no, emp_id(FK), start_date, end_date, leave_type,
    days, reason, status, apply_time, approver_comment
9.6 出勤狀況關聯（供出勤紀錄「出勤狀況」欄）
    可由 makeup_request / leave_request 依 emp_id + 日期關聯彙整，
    或建立檢視（view）回傳當日相關單據與審核狀態。

--------------------------------------------------------------------------------
10. 業務邏輯與規則
--------------------------------------------------------------------------------
10.1 出勤狀態推導
     - 遲到：check_in_state=遲到
     - 早退：check_out_state=早退
     - 缺卡：check_in_state 或 check_out_state=未打卡
     - 上述可同時並存；皆無則為「正常」。
     - 整體 status（正常/異常）：任一班別存在異常節點即為「異常」。
10.2 多班別
     - 同一員工同一日可有多組上/下班；清單與詳情均需逐班別呈現。
10.3 出勤狀況（單據彙整）
     - 依員工＋日期關聯補卡（補卡日期）、請假/銷假（請假期間涵蓋當日）。
     - 顏色：單據狀態=完成→已審核（綠）；其餘→未審核（紅）。
10.4 篩選
     - 多條件 AND；多選下拉為 OR（命中其一即符合）；未選＝不限。
     - 補卡以 makeupDate、出勤/外出以 date、請假以期間重疊判斷日期區間。
10.5 分頁與排序：後端負責分頁計算與排序（預設依日期新到舊）。
10.6 權限與資料範圍
     - 依登入者角色限制可見員工範圍（如僅可見所屬地區/部門/下屬）。
     - 本頁為檢視與審核管理用途；如需審核動作（核准/駁回）由對應審批 API 處理。
10.7 稽核：對匯出、審核等敏感操作記錄操作日誌（操作者、時間、條件）。

--------------------------------------------------------------------------------
11. 非功能性需求
--------------------------------------------------------------------------------
11.1 效能：清單 API 於常見資料量下回應 < 1 秒；匯出大量資料採非同步。
11.2 安全：需身分驗證與授權；防止越權查詢他人資料；輸入參數驗證。
11.3 在地化：繁體中文；日期/時間格式一致。
11.4 可用性：API 失敗時前端顯示友善錯誤提示（Toast）。

--------------------------------------------------------------------------------

--------------------------------------------------------------------------------
12.1 分頁：前端目前為靜態樣式，需確認每頁筆數與後端分頁規格。
12.2 「請假紀錄」分頁：維持 iframe 內嵌或改為原生呼叫請假 API 渲染？
12.3 本頁是否需提供「審核（核准/駁回/要求補件）」動作，或僅檢視？
